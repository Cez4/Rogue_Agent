

🧠 **BRAINSTORM: MELHORES PRÁTICAS RPG + LIMBO AI + COOP LOCAL**


🏗️ **ARQUITETURA RECOMENDADA**

**1. Component-Based Architecture (Entity-Component-System)**
Entity (CharacterBody2D)
├── Components (Scripts standalone):
│   ├── HealthComponent      # Vida/morte, sinais
│   ├── CombatComponent      # Dano, armadura, knockback  
│   ├── StateMachineComponent # HSM ou BT driver
│   ├── InputComponent       # Input routing (P1/P2)
│   ├── NavComponent         # Movement + NavigationAgent
│   └── EmoteComponent       # Emotion bubble
└── Tags/Categories:
    - "player", "npc", "enemy", "boss" via grupos


**2. Behavior Tree Design Pattern**

Princípio: Composição > Herança
BossDragonBT
├── Root: Parallel (Emote + AI)
├── Main: Selector  
│   ├── Sequence: [IsEnraged?] → EnragedBehavior
│   ├── Sequence: [IsUnder30%HP?] → DesperationAttack
│   └── Sequence: [CooldownComplete?] → BasePattern
└── Decorators:
    - Inverter (negar condições)
    - RepeatUntilFail (ataques chanellings)
    - ForceSuccess (transições seguras)

🔧 **Padrões de Código Recomendados**

**1. Interface Segregation para Components**
# Cada component tem interface mínima:
interface IDamageable:
    func take_damage(amount: int, source: Node = null)
    func is_alive() -> bool

interface IMovable:  
    func move_to(target: Vector2)
    func stop()
    func get_velocity() -> Vector2


**2. Event-Driven Communication**
# Em vez de chamadas diretas:
# Ruim: player.get_tree().call_group("npc", "on_player_spotted")

# Bom: 
signal player_spotted(observer, target)
signal player_damaged(victim, amount, source)
signal npc_died(npc, killer)

# Components se conectam via signals


**3. Configuration Over Code**
# Em vez de hardcode:
var walk_speed = 220.0

# Use:
@export var movement_config: MovementConfigResource
# Ou globalmente:
var config = GameConfig.movement.presets["npc_walker"]

🎯 **ANÁLISE DE VIABILIDADE E ESCALABILIDADE**


✅ **VIABILIDADE: ALTO**

O plano é tecnologicamente viável e alinha-se com padrões Godot 4 modernos. Vamos validar:

**Component-Based Architecture**
• ✅ Viável: Godot 4 suporta composition naturalmente via add_node
• ✅ LimboAI já usa composition (HSM + BT Player + Motor)
• ⚠️ Cuidado: Muitos nós filhos podem impactar performance - usar Node simples sem renderização

**Coop Local com Input Abstraction**
• ✅ Viável: Godot tem excelente suporte a múltiplos dispositivos de entrada
• ✅ Singletons são thread-safe no contexto Godot
• ⚠️ Limitação: Input mapping precisa ser configurado manualmente

**Data-Driven Boss System**
• ✅ Viável: Godot 4 Resources são ideais para isso
• ✅ LimboAI suporta instanciação de BT dinâmica
• ⚠️ Cuidado: BT complexos podem causar garbage collection


📈 **ESCALABILIDADE: MODERADA-ATENDE**

**Pontos Fortes**
✓ Component isolation → fácil de otimizar individualmente
✓ BT compartilhado → menos duplicação de código  
✓ Signals over direct calls → decouplamento limpo
✓ Configuration over code → balanceamento sem recompilação


**Gargalos Potenciais**

1. Performance de Behavior Trees  
  • Cada NPC roda seu próprio BT tick
  • 50+ NPCs pode causar lag na atualização
  • Solução: Stagger ticks, caching de condições

2. Sincronização de Estado
  • Cada ação precisa ser replicada
  • Posições/frames de animação precisam sync
  • Solução: Authoritative state, interpolation

3. Gerenciamento de Memória
  • Components são Resources ou Nodes?
  • Instanciação/destruição frequente
  • Solução: Object pooling para NPCs comuns

🎯 **RECOMENDAÇÕES SÊNIOR**

**Se não forescalar muito (≤20 NPCs simultâneos):**
• ✅ Siga o plano atual
• ✅ Simplifique componentes não-críticos
• ✅ Mantenha BTs leves

**Se for escalar (20-100 NPCs):**
MUST HAVE:
├── Spatial partitioning (QuadTree/Grid) para culling
├── LOD system (NPCs distantes têm menos ticks)
├── BT blackboard pooling
└── Animation tree sharing


**Se for coop + muitos NPCs:**
CRITICAL:
├── Deterministic simulation order
├── Fixed timestep networking
├── Input lag compensation
└── Entity interpolation



🚨 **ARMADILHAS COMUNS NO GODOT**

1. Não chame get_tree() em loops - cache a árvore
2. Evite find_node() ou get_nodes_in_group() frequente - use dicionários
3. BT leaf nodes pesados - mova cálculos pesados para _physics_process
4. AnimationPlayer por node - use AnimationTree compartilhado
