# Paper Técnico de Contratos — Rogue Agent V1

Documento técnico para orientar o Tech Lead antes da implementação dos sistemas principais.

Base atual considerada: Rogue Agent V1 em **Godot puro organizado**, **PC 2D**, **coop obrigatório**, **LimboAI já em uso**, **Godot Navigation já testado**, **sem Orchestrator**, **sem GECS**, **sem GECS Network**, **sem GD-Sync**, **sem mobile no V1**. A lógica de Action Queue, Rooms, Smart Objects, Reservation/Slots, Room Events, Task Lifecycle, livros, containers, skills, combate, Adventure Packs e vertical slices foi recuperada das consolidações anteriores do projeto. :contentReference[oaicite:0]{index=0} :contentReference[oaicite:1]{index=1}

# 0. Regras globais de arquitetura

## 0.1 Princípio principal

```txt
Godot puro organizado.
Plugins ajudam.
Sistemas próprios coordenam.
Nenhum plugin vira dono do jogo inteiro.
````

## 0.2 Stack atual

```txt
Engine:
  Godot 2D PC

IA / comportamento:
  LimboAI

Movimento:
  NavigationAgent2D
  NavigationRegion2D

Coop:
  Godot Multiplayer nativo
  NetworkAdapter futuro

Rotina / simulação:
  Action Queue
  Task System
  Smart Objects
  Room Event Bus
  Reservation / Slots

Planejamento:
  GOAP para rotina cotidiana
  Utility scoring simples para prioridades

Gameplay plugins:
  Inventory Manager
  Quest Manager
  Spawner-Godot-Plugin
  Drag-and-drop
  BlastBullets2D

Visual:
  AnimatedSprite2D
  AnimationPlayer
  Phantom Camera
  AnimSheet
  SoftBody2D pontual

Save:
  Addon Save candidato
  SaveManager mínimo se necessário no vertical slice

Testes:
  GUT
```

## 0.3 Removidos / fora da stack atual

```txt
Orchestrator:
  removido

GECS:
  removido/pausado

GECS Network:
  removido

GD-Sync:
  removido

Colyseus/Postgres/Redis:
  upgrade futuro, não V1

Mobile:
  removido do V1

Logic Bricks:
  removido

Cuberact Rope:
  removido

FightEngine:
  plugin fora, conceito absorvido

Save Made Easy:
  referência, mas defasado
```

## 0.4 Regra de autoridade no coop

```txt
Host:
  decide o estado oficial

Clients:
  enviam intenções
  mostram animações/feedback
```

Cliente nunca decide:

```txt
item ganho
inimigo morto
quest completa
task terminada
dano aplicado
drop gerado
HP oficial
```

Cliente pode pedir:

```txt
mover
interagir
atacar
usar skill
arrastar item
aceitar quest
entrar em dungeon
```

# 5. Movement Contract

## Objetivo

Padronizar movimento com Godot Navigation.

## Stack usada

```txt
NavigationRegion2D
NavigationAgent2D
CharacterBody2D
AnimatedSprite2D
LimboAI state
```

## Dados mínimos

```txt
entity_id
current_position
target_position
movement_speed
is_moving
movement_state
navigation_agent_ref
```

## Fluxo click-to-move

```txt
player clica no destino
↓
client/host cria move intent
↓
host valida se destino é navegável
↓
NavigationAgent2D.target_position recebe destino
↓
estado muda para Moving
↓
entity segue get_next_path_position()
↓
ao chegar, estado muda para Idle ou Interacting
```

## Para interação com objeto

```txt
clicar objeto
↓
pegar interaction_point do objeto
↓
mover até interaction_point
↓
ao chegar, iniciar interação/task
```

## Eventos

```txt
move_requested
move_started
move_finished
move_cancelled
move_blocked
```

## Testes

```txt
clicar chão e andar
clicar área inválida e bloquear
clicar objeto e mover até ponto de interação
parar ao chegar
mudar animação walk/idle
```
