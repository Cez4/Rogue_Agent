# 4. Modelo de coop

## 4.1 Coop é pilar do produto

O jogo não é solo-first.
O V1 precisa nascer com coop como objetivo técnico.

```txt
V1:
  2 players

Depois:
  até 4 players
```

## 4.2 Autoridade

```txt
Host:
  decide comportamento oficial
  roda LimboAI
  roda GOAP quando aplicável
  decide combate
  decide task
  decide drops
  decide spawn
  sincroniza estado

Clients:
  enviam intenção
  recebem estado/eventos
  mostram animação/feedback
```

Clientes não devem enviar:

```txt
matei inimigo
ganhei item
completei quest
setei HP
terminei task
```

Clientes enviam:

```txt
quero mover
quero interagir
quero atacar
quero usar skill
quero arrastar item
quero aceitar quest
```

# 7. GOAP

## 7.1 Papel

GOAP entra para planejamento cotidiano do agent.

Não substitui LimboAI.
GOAP define plano; LimboAI executa estados e transições.

Uso:

```txt
fome
sono
leitura
cozinhar
craft
guardar item
pegar item
usar fogueira
usar mesa
usar baú
preparar dungeon
rotina de hideout
sobrevivência
```

Exemplo:

```txt
Objetivo:
  não ficar com fome

Plano:
  procurar comida pronta
  se não tiver, pegar carne
  achar fogueira livre
  cozinhar carne
  esperar ficar pronta
  comer
```

# 12. Reservation / Slots / Locks

Sistema obrigatório para evitar bagunça de múltiplos agentes/NPCs.

## 12.1 Exemplos

```txt
fogueira:
  cook: 1
  sit_near: 2

mesa:
  read: 1
  eat: 1

saco de dormir:
  sleep: 1

baú:
  organize: 1

bancada:
  craft: 1
```

## 12.2 Fluxo

```txt
agent quer usar objeto
↓
verifica objeto
↓
verifica ação
↓
verifica slot
↓
reserva slot
↓
anda até objeto
↓
executa task
↓
libera slot
```

Se ocupado:

```txt
esperar
procurar outro objeto
fazer outra task
conversar
replanejar
```

# 19. Livros e Conhecimento

## 19.1 Livro físico

```txt

não stacka

pode vender
pode trocar
pode guardar
pode ir para estante

```

## 19.2 Livro lido

```txt
vira conhecimento
item é consumido/deletado
não aprende de novo
```

## 19.3 Livro repetido

```txt
não dá conhecimento novamente

```

## 19.4 Leitura

```txt

```

